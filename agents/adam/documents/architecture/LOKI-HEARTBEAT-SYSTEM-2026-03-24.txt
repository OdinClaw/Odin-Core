## HEARTBEAT ARCHITECTURE

Loki's heartbeat is a sophisticated monitoring cycle with 4 layers of resilience:

### Layer 1: Circuit Breaker Gate
**Purpose:** Prevent heartbeat storms and exponential backoff escalation
**Mechanism:** request-circuit-breaker.mjs
**Flow:**
1. Check if previous heartbeat failed
2. If failed, apply exponential backoff before allowing next attempt
3. Backoff schedule:
   - 1 failure → 2 min wait
   - 2 failures → 4 min wait
   - 3 failures → 8 min wait
   - 4 failures → 16 min wait
   - 5 failures → 32 min wait (blocks one 30-min heartbeat window)
   - 6 failures → 64 min wait (blocks two windows)
   - 7+ failures → 60 min max (capped)

**Status:** ✅ Active
**Request ID tracking:** GUID generated for each cycle, used to record outcomes

### Layer 2: Degraded Mode Handler
**Purpose:** Serve heartbeat response when cloud providers unavailable
**Mechanism:** degraded-mode.mjs
**Trigger:** When all cloud providers (Anthropic + Groq) are down or rate-limited
**Response:** Post deterministic heartbeat directly to Discord (no openclaw inference needed)
**Recovery:** Records degraded response as success; allows circuit breaker to reset

**Status:** ✅ Active
**Fallback method:** Direct Discord POST when inference unavailable

### Layer 3a: Groq Throttle Manager
**Purpose:** Enforce rate limits when Groq is fallback provider
**Mechanism:** groq-throttle.mjs
**Rules:**
- Max 1 request every 2 seconds (inter-request spacing)
- Max 20 requests per 60-second sliding window
- Bypass for health probes (GET /models)
- Enqueue deferred requests when throttled
- Auto-drop stale entries (>5 min old)

**Status:** ✅ Active when mode=groq_fallback
**Queue management:** Automatic deferred request processing

### Layer 3b: Gateway Pre-check
**Purpose:** Abort heartbeat if gateway is unreachable
**Check:** HTTP GET /health on gateway (127.0.0.1:18789)
**Timeout:** 4 seconds
**Response:** Skip heartbeat if gateway down (await watchdog restoration)
**Circuit recording:** Marked as SUCCESS (not a provider failure)

**Status:** ✅ Active
**Port:** 18789 (configurable via OPENCLAW_GATEWAY_PORT)

### Layer 4: Openclaw Invocation
**Purpose:** Main heartbeat logic (check cron, validate jobs, report to Discord)
**Command:** openclaw --profile odin agent --agent loki [message] [channel] [timeout]
**Timeout:** 60 seconds hard limit (kill after 60s + 5s grace + 9s force)
**Retries:** Max 2 retries on failure (3 total attempts)
**Outcome recording:** Success/failure logged to circuit breaker

**Status:** ✅ Operational
**Message:** "Read HEARTBEAT.md. Check cron list. For each failed job: report name, error status, last error message, and specific fix. If all OK, post '✅ Heartbeat OK'. Post findings to #loki."

## HEARTBEAT TRIGGER FLOW

```
1. Cron fires trigger-heartbeat.sh
   ↓
2. Circuit breaker gate: check if backoff active
   ├─ YES → SKIP (backoff active, exit 0)
   └─ NO → continue with requestId
   ↓
3. Degraded mode check: all providers down?
   ├─ YES → post deterministic response, mark SUCCESS, exit 0
   └─ NO → continue
   ↓
4a. Groq throttle check (if groq_fallback mode)
    ├─ THROTTLED → enqueue request, mark SUCCESS, exit 0
    └─ SLOT OK → continue
   ↓
4b. Gateway pre-check: gateway reachable?
    ├─ NO → SKIP (mark SUCCESS), exit 0
    └─ YES → continue
   ↓
5. Invoke openclaw (loki heartbeat)
   ├─ SUCCESS → record SUCCESS, exit 0
   └─ FAILURE → apply backoff, exit 0
```
